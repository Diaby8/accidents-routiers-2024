JUSTIFICATION OF DESIGN CHOICES
TP Data Integration & Applications — ST2DLDI
Dataset: Road traffic injury accidents, France 2024 (data.gouv.fr)
======================================================================

1. DROPPING NEAR-EMPTY COLUMNS
----------------------------------------
Columns lartpc (100% empty), v2 (91.6% empty) and occutc (99% empty) were
dropped as early as the Silver layer. A column with almost no data carries no
usable information and needlessly bloats the dataset. Keeping them could have
been misleading during analysis.

2. HANDLING DOM-TOM
--------------------------
The 3,344 accidents whose department code is 971-978 or 986-988 correspond to
overseas territories (Martinique, Reunion, Guyane, etc.). These are not
errors: they are genuine, valid data. The split is done on the department
code rather than a latitude threshold: a threshold (lat < 41 deg or lat > 52
deg) is unreliable, it wrongly classifies Saint-Pierre-et-Miquelon (dep. 975,
latitude ~46-47 deg, within the mainland range) as mainland France, and can
classify southern Corsica (dep. 2B) as DOM-TOM. We chose to set these rows
aside in a separate file rather than delete them, in order to scope the
analysis to mainland France while keeping this data available for a future
study.

3. REPLACING -1 VALUES WITH NaN
----------------------------------------
In this dataset, -1 is used as a code meaning "not specified". Left as-is,
every statistical computation (averages, distributions) would be biased
because Python would treat it as a real numeric value. Replacing it with NaN
(Not a Number) is the standard convention for representing an unknown value
and lets pandas functions ignore it automatically.

4. IMPUTING "UNKNOWN" ON MISSING TEXT FIELDS
-----------------------------------------------------------
For columns adr (address, 4.2% empty) and voie (road number, 19% empty), we
chose to impute "Unknown" rather than drop the affected rows. Dropping an
entire row because of a missing address would discard every other piece of
information about that accident (date, severity, weather, etc.) that remains
usable.

5. DATA ENRICHMENT
------------------------------
Two columns were added because they don't exist in the raw data but are
directly useful for analysis:

- time_of_day: buckets the accident hour into 4 categories (Morning,
  Afternoon, Evening, Night). More readable than a raw hour for identifying
  at-risk periods.

- max_severity: indicates the most severe outcome among all users involved in
  the same accident. Lets each accident be characterized by a single,
  representative figure without having to join the users table on every
  query.

6. CHOICE OF A STAR SCHEMA FOR THE GOLD LAYER
--------------------------------------------------
Three models were considered for organizing the Gold layer:

- Flat table: everything in a single table. Simple but full of duplicates
  since an accident involves multiple vehicles and multiple users.

- Snowflake schema: dimensions are themselves broken down into sub-tables
  (e.g. dim_location -> dim_department -> dim_region). More normalized but
  more complex to query, and justified only for very large datasets.

- Star schema: a central fact table linked to dimension tables. This is the
  option chosen.

We chose the star schema because:
  - It fits the real complexity of the data (one accident = several
    vehicles and users, hence separate tables)
  - It performs well for BI analysis (few joins needed)
  - It matches exactly what the assignment suggests: "fact table + dimensions"
  - It is simpler to maintain and understand than a snowflake schema for this
    use case

The join key across every table is Num_Acc (unique accident number), which
makes it easy to reconstruct a complete view of any accident.
